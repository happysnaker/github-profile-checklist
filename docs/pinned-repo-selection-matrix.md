# Pinned Repo Selection Matrix

Use this matrix when deciding which repositories deserve a slot in your six pinned items.

The point is not to pin your “favorite” repos.
The point is to pin the repos that best support the story you want a stranger to believe.

---

## Step 1: Score candidate repos

Rate each repo from 0–3 in the categories below.

### Category guide

#### Role fit

- **3** = directly supports the role you want next
- **2** = adjacent and still helpful
- **1** = loosely related
- **0** = confuses your story

#### Technical depth

- **3** = demonstrates real implementation depth
- **2** = shows solid engineering judgment
- **1** = useful but shallow
- **0** = mostly cosmetic / weak

#### Reusability

- **3** = other people could actually use it
- **2** = useful as a reference
- **1** = mostly personal
- **0** = hard to justify publicly

#### README quality

- **3** = strong first impression
- **2** = understandable but could improve
- **1** = weak or unclear
- **0** = actively hurts the repo

#### External proof

- **3** = stars / forks / adoption / contribution visibility
- **2** = some visible social proof
- **1** = little proof
- **0** = no credibility support yet

#### Current quality

- **3** = feels intentional and maintained
- **2** = decent but rough edges remain
- **1** = obviously old or messy
- **0** = should probably not be pinned

---

## Step 2: Build a balanced pin set

A strong six-repo set often includes a mix like this:

1. **depth repo** — something technically serious
2. **build repo** — something you implemented end-to-end
3. **reusable repo** — starter, middleware, template, toolkit
4. **systems / infra / architecture signal**
5. **teaching / explanation / notes repo**
6. **social-proof repo** — stars, adoption, or visible reuse

You do **not** need one repo for each bucket, but your set should not feel one-dimensional.

---

## Step 3: Final pin questions

Before pinning a repo, ask:

- Would this help or hurt if an interviewer clicked it first?
- Does the README explain why it exists in under 20 seconds?
- Does it support the role I want next?
- Does it make the whole set more coherent?
- Would I still pin this if it had zero sentimental value?

If the answer to the last question is “no”, it probably should not be pinned.

---

## Example pin sets

### Backend / infra / systems engineer

- database / storage implementation repo
- RPC / networking / concurrency repo
- reusable Go service starter
- reusable middleware / library
- system design / production checklist
- one repo with strong stars or adoption

### Frontend / product engineer

- strongest shipped UI project
- component system / design system
- polished product-like app
- accessibility / responsiveness proof
- one teaching repo or technical writeup
- one repo with stars / reuse / social proof

### Student / early-career engineer

- best implementation project
- best reusable asset
- best systems / backend / frontend match for target role
- one notes / learning repo that teaches something clearly
- one external open-source contribution
- one repo with visible reuse / stars if available
