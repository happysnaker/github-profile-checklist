# Before / after example: making a backend GitHub profile feel more credible

This is a compact example of how a noisy public GitHub can be turned into a more technical, role-aligned profile.

## Before

Typical weak signals:

- bio is vague or student-only
- pinned repos are random or outdated
- strongest repo is buried under toy projects
- old coursework / experiments dominate first impressions
- README files are thin and do not explain why the repo exists
- all public work looks local to one person, with no outside contribution signal

### What a hiring manager may infer

- unclear engineering direction
- weak prioritization
- unfinished or accidental public presence
- not much evidence of taste, packaging, or cleanup ability

## After

Typical strong signals:

- one-line bio clearly states target role and technical area
- pinned repos support the same story
- weak / random projects are hidden, archived, or simply not foregrounded
- top repos have readable descriptions, topics, and README files
- at least one public repo feels reusable, not merely personal
- there is some external proof of work such as focused OSS bug fixes
- contact / portfolio / review path is obvious but not spammy

### What a hiring manager may infer

- knows what kind of engineer they are
- can finish and package work
- understands first impressions and technical storytelling
- has enough implementation signal to warrant an interview

## Good transformation pattern for backend / systems candidates

1. fix bio and website link
2. choose 4-6 pinned repos that match the target role
3. move weak old projects out of first-impression space
4. rewrite the top README files first
5. add one or two credible external code contributions
6. only then optimize visual polish or profile cosmetics

## When a checklist is enough

A checklist is usually enough when:

- you already know your target role
- you can tell which repos are weak
- you just need structure and prioritization
- your README files are fixable with normal editing effort

## When a review is faster

A paid async review is often faster when:

- you cannot tell which repos are helping or hurting
- your GitHub still feels "student" or "random" even after cleanup
- you want a second opinion on repo order, README framing, or profile positioning
- you want ready-to-paste edits instead of only guidance
